# RELAY_PROTOCOL (Русский)

Последнее обновление: 2026-04-20
Статус: черновик, согласованный с реализацией

Документ описывает relay-контракт в том виде, в котором его сегодня использует текущий runtime.

## 1. Назначение

Определяет relay-контракт доставки сообщений, который использует текущий runtime.

## 2. Роли

- Узел клиента: хранит/получает/подтверждает конверты.
- Сервер relay: хранит зашифрованные/подписанные конверты и обслуживает API выборки/ack.

## 3. Модель конверта

Клиент отправляет конверт relay с полями:
- `id`
- `from`
- `to`
- `timestampMs`
- `ttlSeconds`
- `payload` (байты)
- `signature`
- `senderSigningPublicKey`

Поведение во время выполнения:
- сам полезный груз конверта содержит надежный конверт (`plainMessage`, `secureMessage`, типы handshake),
- путь приема проверяет подпись перед обработкой.
- сервер relay проверяет подписи на endpoints API relay (`store`, `group/store`, `group/members/update`, `ack`, загрузка blob/finalize).

Полезный груз подписи для проверки конверта:

- `id|from|to|timestampMs|ttlSeconds|` + сыр `payload` байты

## 4. HTTP API (используемый клиентом)

- `POST /relay/store` — хранить конверт
- `POST /relay/group/store` — хранить один подписанный конверт группы и fan-out к перечисленным получателям
- `POST /relay/group/members/update` — обновить авторитетное членство группы на relay
- `GET /relay/fetch?to=<peerId>&limit=<n>&cursor=<optional>` — получить конверты
- `POST /relay/ack` — подтвердить доставку конверта
- `POST /relay/blob/upload` — загрузить зашифрованный полезный груз blob (one-shot)
- `GET /relay/blob/:blobId` — получить полезный груз blob по id
- `POST /relay/blob/upload/chunk` — загрузить один chunk большого blob
- `POST /relay/blob/upload/complete` — завершить chunked upload и материализировать blob

Тело запроса `/relay/ack` теперь включает:

- `id`
- `from`
- `to`
- `timestampMs` (`ts`)
- `signature` (`sig`)
- `senderSigningPublicKey` (`signingPub`)

Полезный груз подписи Ack:

- `id|from|to|timestampMs`

Полезный груз подписи загрузки Blob:

- `id|from|groupId|fileName|mimeType|timestampMs|ttlSeconds|` + сыр полезный груз blob байты

Примечание по direct media:

- личное медиа использует тот же контракт blob upload,
- клиент выводит детерминированный direct blob scope в поле `groupId`: `dm:<peerA>|<peerB>` (peer id сортируются),
- сама доставка личного медиа в чат выполняется через зашифрованный `direct_blob_ref`, передаваемый по обычному каналу reliable personal message.

Полезный груз подписи обновления членства группы:

- `id|from|groupId|ownerPeerId|member1,member2,...|timestampMs|ttlSeconds`

## 5. Обмен сообщениями группы через Relay

Текущая группа runtime использует relay как источник истины:

- текст группы и медиа группы доставляются через путь relay группы,
- медиа группы загружается один раз как blob и распространяется через метаданные сообщения группы,
- зашифрованный полезный груз медиа группы упакован в компактном двоичном формате (`PLG2`) с резервной декодировкой на стороне клиента,
- большие криптографические операции медиа группы выполняются в фоновом изолированном процессе на стороне клиента избегая зависаний UI,
- для больших blob клиент сначала пытается chunked upload (`/chunk` + `/complete`), затем откатывается до one-shot `/relay/blob/upload` если API chunk недоступен.
- членство группы синхронизируется владельцем группы через `/relay/group/members/update`,
- relay применяет членство на пути записи (`/relay/group/store` и endpoints загрузки blob группы),
- ключ группы ротируется при изменении членства (добавление/удаление участников).

## 5.1 Личное медиа через Relay

Текущий runtime для личного медиа использует relay как blob-хранилище:

- payload файла один раз загружается через `/relay/blob/upload` (или через chunked blob upload, если он доступен),
- затем sender отправляет зашифрованный `direct_blob_ref` по обычному пути personal relay message,
- получатель резолвит blob reference и восстанавливает медиа из relay blob storage,
- прием личного медиа теперь работает только через зашифрованный `direct_blob_ref` и загрузку blob из relay; legacy-путь `fileMeta/fileChunk` удален.

## 6. Поведение с несколькими серверами

Клиент хранит список настроенных relay, но runtime больше не работает со всеми серверами как с одинаково активными.

Текущее поведение при нескольких relay:
1. сначала выполняется быстрый health-aware отбор,
2. для runtime-операций предпочитаются только живые relay,
3. активный рабочий набор остается небольшим (до 3 relay),
4. для записи и ack используется quorum,
5. операции по выбранным relay выполняются параллельно, чтобы один dead relay не добавлял полную последовательную задержку.

Стабилизация, специфичная для выборки:
- клиент GET fetch использует переходный retry/backoff для ошибок транспорта connection-drop/timeout перед отметкой сервера как нездорового.

Заметка эксплуатации:
- при timeout/http ошибках для всех серверов клиент деградирует мягко, без hard crash,
- статус relay показывается в UI через `NodeFacade`,
- `HttpRelayClient` пишет явный runtime log, когда relay отвечает `401 invalid signature`.

## 7. Заметки безопасности

- Конверты подписаны и проверены при приеме.
- Сервер relay проверяет подписи для store/group-store/group-members-update/ack/blob upload/finalize.
- Сервер relay применяет членство на endpoints записи группы.
- Проверки replay применяются при обработке надежного конверта.
- Путь шифрования reliable-сообщений в runtime активен (`enableEncryption: true`).

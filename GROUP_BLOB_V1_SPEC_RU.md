# Group Blob Messaging v1 (Спецификация)

Последнее обновление: 2026-04-12
Статус: частичная реализация. Текущий runtime уже использует `/relay/group/store` и blob upload path, а проверки членства на стороне сервера и ротация group-key уже активны.

## 1. Цель

- Использовать одну модель доставки для **текста группы** и **медиа группы**.
- Минимизировать нагрузку на устройство отправителя: `загрузить один раз + отправить одно событие группы`.
- Сохранить приватность E2E с помощью шифрования ключом группы.
- Подготовиться к пробуждению по push и сборке мусора blob.

---

## 2. Область применения v1

В области:

- Транспорт обмена сообщениями группы (`blob + событие группы`).
- Конвейер выборки клиента по seq.
- Контракт пробуждения по push (через сервер начальной загрузки).
- Политика TTL blob и сборки мусора.

Вне области:

- Полная архитектура разделения тем/мультирегиона.
- История ключей на длительный срок, кроме активного + предыдущего ключа.

---

## 3. Модель данных

### 3.1 Событие группы

- `eventId: string`
- `groupId: string`
- `seq: int64` (назначено сервером)
- `senderPeerId: string`
- `kind: "text" | "media" | "group_key_update"`
- `blobId: string?`
- `ts: int64` (временная метка клиента)
- `ttl: int`
- `sig: base64`
- `signingPub: base64`

### 3.2 Blob

- `blobId: string`
- `groupId: string`
- `fileName: string`
- `mimeType: string?`
- `sizeBytes: int`
- `payload: base64` (зашифрованный полезный груз)
- `ts: int64`
- `ttl: int`
- `sig: base64`
- `signingPub: base64`

## 4. Контракт API v1 (целевая версия)

### 4.1 `POST /relay/blob/upload`

Назначение:

- Загрузить зашифрованный полезный груз один раз.

Тело запроса JSON:

```json
{
  "id": "blob:1770000000000",
  "from": "peerA",
  "groupId": "group:123",
  "fileName": "photo.jpg",
  "mimeType": "image/jpeg",
  "ts": 1770000000000,
  "ttl": 2592000,
  "payload": "base64...",
  "sig": "base64...",
  "signingPub": "base64..."
}
```

Ответ JSON:

```json
{
  "ok": true,
  "blobId": "blob:1770000000000"
}
```

### 4.2 `GET /relay/blob/:blobId`

Назначение:

- Загрузить полезный груз blob по ID.

Ответ JSON:

```json
{
  "id": "blob:1770000000000",
  "groupId": "group:123",
  "fileName": "photo.jpg",
  "mimeType": "image/jpeg",
  "sizeBytes": 245667,
  "payload": "base64..."
}
```

### 4.3 `POST /relay/group/event`

Назначение:

- Добавить одно событие в журнал группы.

Тело запроса JSON:

```json
{
  "eventId": "1770000001111",
  "groupId": "group:123",
  "senderPeerId": "peerA",
  "kind": "media",
  "blobId": "blob:1770000000000",
  "ts": 1770000001111,
  "ttl": 2592000,
  "sig": "base64...",
  "signingPub": "base64..."
}
```

Ответ JSON:

```json
{
  "ok": true,
  "seq": 1042
}
```

### 4.4 `GET /relay/group/events?groupId=...&afterSeq=...&limit=...`

Назначение:

- Получить события группы после последней локальной последовательности.

Ответ JSON:

```json
{
  "groupId": "group:123",
  "events": [
    {
      "eventId": "1770000001111",
      "seq": 1042,
      "senderPeerId": "peerA",
      "kind": "media",
      "blobId": "blob:1770000000000",
      "ts": 1770000001111
    }
  ],
  "lastSeq": 1042
}
```

### 4.5 `POST /relay/group/ack` (опионально v1)

Назначение:

- Клиент подтверждает локальную обработку до `seq`.

## 5. Конвейер клиента v1 (целевая версия)

### 5.1 Отправить текст (группа)

1. Зашифровать байты текста текущим ключом группы (`AES-GCM-256`).
2. `POST /relay/blob/upload`.
3. `POST /relay/group/event` с `kind=text`, `blobId`.
4. Отметить локальное сообщение как отправленное после успеха `group/event`.

### 5.2 Отправить медиа (группа)

1. Зашифровать байты медиа текущим ключом группы.
2. `POST /relay/blob/upload`.
3. `POST /relay/group/event` с `kind=media`, `blobId`, метаданные.
4. Отметить локальное сообщение как отправленное после успеха `group/event`.

### 5.3 Получить

1. Получить/выбрать `group/events` из `afterSeq`.
2. Для каждого события с `blobId` получить blob один раз.
3. Расшифровать полезный груз через ключ группы.
4. Сохранить в локальное хранилище и обновить UI.

### 5.4 Повтор/идемпотентность

- `blobId` и `eventId` генерируются клиентом и являются идемпотентными.
- Экспоненциальный backoff: `1,2,4,8,16,30s`.
- При переподключении: возобновить по `afterSeq`.

---

## 6. E2E ключ группы v1

- Шифр: `AES-GCM-256`.
- Один активный ключ на группу + один предыдущий ключ.
- Ротировать ключ при изменении членства.
- Обновление ключа передается как событие `group_key_update`.
- Полезный груз ключа шифруется по-отдельно для каждого получателя на открытом ключе получателя.

---

## 7. Контракт пробуждения по push v1 (сервер начальной загрузки)

Назначение:

- Пробудить клиент; содержимое сообщения все еще извлекается из relay.

FCM полезный груз `data`:

```json
{
  "type": "group_update",
  "groupId": "group:123",
  "lastSeq": "1042",
  "unreadDelta": "3",
  "ts": "1770000002222"
}
```

Правила:

- Отправлять push только для офлайн/неактивных пользователей.
- Дебаунс по `(peerId, groupId)` на 2-5 секунд.
- Клиент должен выбрать через `group/events`; push только подсказка, не источник истины.

---

## 8. Политика GC Blob v1

- TTL blob по умолчанию: `30 дней`.
- Интервал GC: `10-30 мин`.
- Сохранить окно безопасности (не удалять очень свежие blob даже если часы дрейфят).
- Если blob истек, но событие существует, отметить событие как истекшее в UI клиента.
